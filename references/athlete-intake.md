# Athlete Intake

What to collect before programming a HYROX athlete, and how much each field
actually changes the output. Intake is the input to every other skill in this
pack: a plan, projection, or pacing model is only as good as the assessment it
rests on. Collect what materially changes the answer; do not interrogate the
athlete for data you will not use.

**Privacy.** Intake data is used in-session only to produce the requested
coaching output. It is not stored, transmitted to a third party, or used to
train anything. Say this plainly if the athlete asks. Never fabricate intake
data to fill a gap — name the gap and label the answer provisional (see
[safety-and-referral](safety-and-referral.md)).

## High-leverage fields (these materially change outputs)

These move the plan, the projection, or the pacing model enough that a wrong or
missing value produces a meaningfully wrong answer.

- **Race date.** Sets the number of training weeks and therefore the entire
  periodization, the taper window ([tapering](tapering.md)), and how aggressive
  a goal is realistic.
- **Division.** Singles/Pro/Doubles/Relay and age group drive loads, standards,
  and partner logic. Loads and standards are **season-dependent and versioned** —
  collect the division so the rules advisor can resolve the right season; never
  assume the weight.
- **Goal time (and how it was set).** A goal anchored to a prior finish is
  usable; an aspirational number with no basis gets flagged as such. Distinguish
  "I ran 1:12 last season" from "I want sub-60 because it sounds good."
- **Training availability** — days per week and **session length**. The single
  biggest realism constraint. An elegant plan the athlete cannot complete is a
  bad plan (see [coaching-principles](coaching-principles.md), §8).
- **Running background.** Recent weekly mileage, easy/long-run pace, any
  threshold or 5 km/10 km benchmark, injury history with running. This anchors
  the [running-development](running-development.md) progression and the fresh
  pace that the split model needs.
- **Strength background.** Training age, recent loads on hinge/squat/press/carry,
  comfort under load. Drives sled and station prescriptions and how fast they can
  progress.
- **Station proficiency.** Per-station comfort and any known limiters
  (technique, grip, capacity) across the 8 stations — see
  [station-taxonomy](station-taxonomy.md). Identifies where the time-loss
  profile likely sits.
- **Injury / health restrictions.** Current pain, prior injuries, surgeries,
  medical conditions, anything a clinician has flagged. Any red-flag content here
  routes to [safety-and-referral](safety-and-referral.md) before any programming.

## Useful context fields

Sharpen the plan but rarely flip the headline answer on their own.

- **Recent performances.** Race results, time-trial splits, recent simulation
  data ([simulation-design](simulation-design.md)). The best calibration source
  when present.
- **Concurrent training.** Other sports, CrossFit, lifting programs, classes the
  athlete will not give up. These compete for recovery and must be sequenced, not
  ignored.
- **Life / recovery load.** Sleep, work stress, shift patterns, travel,
  nutrition habits, age. Sets how much load is recoverable and how conservative
  the progressions should be.
- **Equipment access.** Sled, ergs, sandbags, wall-ball targets, treadmill vs
  outdoor running, gym vs home. Decides what can be prescribed directly and what
  needs substitution.
- **Partner data** (Doubles/Relay only). Each partner's running, station
  strengths, and fatigue profile — see [doubles-strategy](doubles-strategy.md)
  and [relay-strategy](relay-strategy.md).

## Optional / nice-to-have

- Preferred running surface, music/cadence habits, prior coaching, motivation and
  goal beyond this race. Improves adherence and tone; does not change the math.

## How intake feeds the rest of the pack

| Field | Primarily feeds |
|-------|-----------------|
| Race date | periodization, [tapering](tapering.md) |
| Division | rules/loads advisor, [pacing-model](pacing-model.md) |
| Running background | [running-development](running-development.md), fresh pace in the split model |
| Station proficiency | [station-taxonomy](station-taxonomy.md), priority ranking |
| Availability / session length | overall plan feasibility |
| Injury / health | [safety-and-referral](safety-and-referral.md) routing |

## Handling missing data

When a high-leverage field is missing, do not invent it. State the assumption
explicitly (e.g. "assuming a ~5:00/km fresh easy pace, typical for a 22:00
5 km"), label the output provisional, and widen any projection's confidence band.
Ask for the one or two fields that would most tighten the answer rather than
demanding a full questionnaire.
