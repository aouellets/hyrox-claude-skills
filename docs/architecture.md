# Architecture

This repository is a **standalone, open-source content source**: 27 HYROX coaching
skills, the tooling that validates them, and a deterministic seed pipeline that
feeds the **Skill Me** catalog. It is not official, endorsed, sponsored, approved,
or certified by HYROX World GmbH or its affiliates (see
[branding-and-trademarks.md](branding-and-trademarks.md)).

Nothing here runs at request time. The repo produces two artifacts —
`skill-inventory.json` and `seed/seed.sql` — from the skills, and that is the
entire output. The catalog (a separate Next.js + Supabase + Vercel app) ingests
those artifacts and serves the skills over MCP.

## Data flow

The content flows in one direction, from human-authored `SKILL.md` files to the
catalog's Supabase tables, then out over MCP:

```
skills/<id>/SKILL.md  (frontmatter + body, authored by hand)
        │
        │  scripts/build-inventory.ts   (npm run build:inventory)
        ▼
skill-inventory.json  (GENERATED — { pack, skill_count, skills[] })
        │
        │  scripts/build-seed.ts        (npm run build:seed)
        ▼
seed/seed.sql         (GENERATED — idempotent upsert, wrapped in begin/commit)
        │
        │  psql / Supabase SQL editor   (maintainer applies, see installation.md)
        ▼
Supabase: public.skills · public.packs · public.pack_skills
        │
        │  Skill Me MCP server (https://skillshelf-ten.vercel.app/api/mcp)
        ▼
browse_skills · install_skill · install_pack · get_active_skills
```

Key properties of the pipeline:

- **`build-inventory.ts` reads only frontmatter.** It emits `skill-inventory.json`
  with `{ id, name, description, category, risk_level, requires_current_rules }`
  per skill. This is the single intermediate artifact.
- **`build-seed.ts` never reads `SKILL.md` directly.** It consumes
  `skill-inventory.json` plus `manifest.json`, and reads each `SKILL.md` only to
  copy its full text into `skills.skill_content` (the installable payload). This
  keeps the seed's view of metadata consistent with the inventory.
- Both generated files carry an `AUTOGENERATED ... do not edit by hand` header.
  Regenerate them with `npm run build`; do not hand-edit.

### What lands in Supabase

The seed targets the live skillshelf schema, **confirmed via the Supabase MCP
`list_tables` on 2026-06-21** (re-confirm before targeting any other project):

- `skills.id` is a **server-generated uuid** (`gen_random_uuid()`); the stable
  natural key is the **UNIQUE `slug`**. The upsert conflict target is therefore
  `(slug)`, and `pack_skills` rows resolve uuids by joining on slug.
- `category` is a **fixed enum**
  (`coding | writing | research | productivity | data | design | business | personal[,mixed]`).
  Fitness has no native bucket, so **every skill maps to `personal`**, and the
  HYROX domain category (e.g. `hyrox-running`) is carried in `subcategory`.
- `skills.skill_content` stores the installable `SKILL.md` (frontmatter + body).
- Because the pack is not brand-authorized, `verified` is `false`, `author` is the
  neutral `hyrox-claude-skills`, and the pack uses the neutral display name
  "Fitness Racing Performance Pack" (see
  [branding-and-trademarks.md](branding-and-trademarks.md)).

The one genuine placeholder in the generated SQL is `REPO_URL` — replace `OWNER`
with the real GitHub org/repo before publishing (see
[installation.md](installation.md)).

## `lib/` determinism and privacy

`lib/goal-time-model.ts` and `lib/split-analysis.ts` hold the only math in the
repo. By design they:

- make **no network calls**,
- read **no clock** and depend on no ambient state,
- collect and persist **no athlete data**, and emit **no telemetry**.

They are pure functions over their inputs, so the same inputs always produce the
same output and the unit tests in `tests/` can pin exact numbers. `goal-time-model.ts`
exports the race model (`STATION_IDS`, `computeRaceTime`, `projectFinishRange`,
`allocateSplitsForGoal`, `formatHMS`/`parseHMS`); `split-analysis.ts` exports the
post-race math (`analyzePaceDecay`, `comparePlannedVsActual`, `summarizeTopLoss`).
Skills that quote split math describe the model in prose so an athlete can audit it
rather than trusting an opaque number. No athlete data is ever stored anywhere in
this repo or the seed.

## The validation gate and how the pieces stay consistent

`scripts/_lib.ts` is the **single source of truth** for what a valid skill looks
like. It exports the contract constants that every validator and build script
imports, so they can never disagree:

- `REQUIRED_FRONTMATTER` — the frontmatter keys every `SKILL.md` must declare.
- `RISK_LEVELS` — allowed `risk_level` values (`low | moderate | high`).
- `DOMAIN_CATEGORIES` — the ten in-repo domain categories (distinct from the DB
  enum the seed maps to `personal`).
- `REQUIRED_SECTIONS` — the H2 headings every `SKILL.md` body must contain.
- `MIN_EXAMPLES`, `MIN_EVAL_CASES`, `REQUIRED_CASE_KEYS` — the eval-suite shape.

It also provides the shared frontmatter parser, skill discovery (`listSkillIds`,
`loadAllSkills`), and console/reporting helpers.

`npm run validate` runs the five gates in order:

1. **`validate:frontmatter`** — frontmatter is present, well-typed, and `name`/id
   are unique.
2. **`validate:skills`** — folder structure (every required H2 section, an H1, a
   `examples/` dir with `>= MIN_EXAMPLES` files, `evals/` with `cases.json` +
   `rubric.md` + `expected-behaviors.md`, a well-formed `cases.json` with
   `>= MIN_EVAL_CASES` cases and `>= 1` adversarial case). It also checks that
   rules-dependent skills (`requires_current_rules: true`) actually reference a
   season/version requirement in their body.
3. **`validate:links`** — every relative / `references/` / `lib/` / `skills/` link
   in every markdown file resolves to a real file (external links and anchors are
   skipped), so the routing and reference web never rots.
4. **`validate:triggers`** — `description` frontmatter is trigger-precise: long
   enough, free of banned filler, and mentions HYROX.
5. **`validate:sources`** — `source-version-registry.json` entries have the
   required metadata and valid dates, and `brand-authorization.json` is internally
   consistent (it cannot claim official/verified while `authorized` is `false`).

Because every gate and both build scripts import the same constants from `_lib.ts`,
the inventory, the seed, and the validators always share one definition of "valid".
`npm run check` runs the whole chain — typecheck, validate, test, build, evals —
and is the single command CI and contributors should treat as the gate.
